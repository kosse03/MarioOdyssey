# H3. 굼바 스택 재귀 구조

## 1. 네 가지 대표가 다르다

굼바 스택이 어려운 이유는 스택 하나에 대표 객체가 네 종류이기 때문이다.

| 대표 | 객체 | 이유 |
|---|---|---|
| 물리 대표 | Root | 바닥과 접촉하고 실제 이동 |
| 캡처 대표 | Top | 캐피가 캡처할 대상 |
| 입력 대표 | Captured Member | PlayerController가 Possess |
| 피격 대표 | Captured Member 또는 Root | 플레이어 리액션 전달/물리 넉백 |

이 네 역할을 혼동하면 입력은 위에 들어가는데 아래가 움직이지 않거나, 아래를 때렸는데 캡처 이벤트가 오지 않는 문제가 생긴다.

## 2. 링크 불변식

정상 체인에서 항상 다음이 맞아야 한다.

```text
Node.Above != null 이면 Node.Above.Below == Node
Node.Below != null 이면 Node.Below.Above == Node
Root.Below == null
Top.Above == null
한 Node를 Above 방향으로 순회했을 때 다시 자기 자신을 만나지 않음
```

WeakObjectPtr는 파괴된 Actor의 수명 문제는 줄이지만 링크를 자동으로 재연결하지는 않는다. 중간 노드가 Destroy되면 위·아래 이웃을 연결하거나 스택을 두 개로 분리하는 정책이 필요하다.

## 3. 두 스택 결합

위 스택 A가 아래 스택 B에 착지할 때 실제 연결 지점은 다음이다.

```text
B.Top.Above = A.Root
A.Root.Below = B.Top
```

A의 충돌한 개별 굼바와 B의 충돌한 개별 굼바를 그대로 연결하면 중간 노드끼리 이어져 체인이 갈라질 수 있다. 반드시 각 스택의 경계 노드를 Resolve해야 한다.

또한 다음을 먼저 검사해야 한다.

- A와 B의 Root가 같은가: 이미 같은 스택
- B가 A의 Above 체인에 있는가: 순환 생성
- A 스택에 Captured Member가 있는가: 플레이어 주도 스택만 결합
- B 대상이 실제 Top인가

## 4. Follower Transform

Follower에 물리를 그대로 두면 Capsule들이 서로 밀어 스택이 폭발하듯 흔들린다. 그래서 Root만 CharacterMovement를 사용하고 Follower는 목표 높이로 정렬한다.

목표 Z 간격은 각 굼바 캡슐/설정에 맞춰 계산돼야 한다. 스택이 회전할 때 모든 Follower가 Root Yaw를 따라야 시각적으로 한 몸처럼 보인다.

Tick 정렬의 위험은 프레임 순서다. 각 노드가 자기 Below를 기준으로 동시에 갱신하면 아래쪽부터 위쪽 순서가 아닐 때 한 프레임 지연이 누적될 수 있다. Root가 전체 Above 체인을 한 번에 순서대로 배치하는 방식이 더 결정적이다.

## 5. 입력 전달

Captured Top이 입력을 받으면 Root를 Resolve하고 Root Movement에 적용한다.

```text
CapturedMember.InputMove
  → Root = FindStackRoot()
  → 카메라 Yaw 기준 방향 계산
  → Root.AddMovementInput

CapturedMember.InputJump
  → Root.Jump
```

속도 파라미터는 Captured Member의 설정을 Root에 적용한다. 따라서 Root Blueprint가 다른 굼바 변형이어도 플레이어가 캡처한 타입의 조작감을 유지할 수 있다.

해제 후에는 Root에 임시 적용한 속도와 입력 상태를 원래 AI 기준으로 되돌려야 한다.

## 6. 피해 전달의 재귀 방지

스택 어느 층이 맞아도 Captured Member로 피해를 전달하지만, Captured Member 자체가 다시 같은 전달 로직을 타면 무한 재귀가 생길 수 있다.

안전한 규칙은 다음과 같다.

- 현재 `this == CapturedMember`면 다시 ApplyDamage하지 않고 Super 처리
- 다른 노드만 CapturedMember에 ApplyDamage
- 전달된 피해의 DamageCauser/Instigator를 유지
- CaptureComponent가 한 번만 OnTakeAnyDamage를 받는지 검증

현재 구조는 캡처 대상에 피해를 모아 피격 Delegate를 발생시키는 의도다. HP 전달 문제와 별개로 재귀/중복 횟수 테스트가 필요하다.

## 7. 보상 시스템 연결

Goombette는 체인을 직접 검색하지 않고 현재 PlayerPawn인 굼바의 `GetStackCount()`만 호출한다. 이 API가 어느 노드에서 호출돼도 전체 스택 수를 반환해야 한다.

```text
PlayerPawn == Overlap Actor
AND PlayerPawn is Goomba
AND StackCount >= 4
→ SuperMoon Spawn
```

이 구조는 AI 굼바 스택이 우연히 Trigger에 들어오는 것을 막는다. 반면 스택 Root의 Capsule만 실제 Overlap을 만들고 PlayerPawn은 Top이면 `OtherActor != PlayerPawn` 조건 때문에 트리거가 발동하지 않을 수 있다. 현재 충돌 설정에서 Top도 Query Overlap을 생성하는지 PIE에서 반드시 검증해야 한다.

## 8. 테스트용 불변식 검사

Development 빌드에서 다음 검사를 유틸리티로 만들면 디버깅이 쉬워진다.

```text
ValidateStack(Node):
  Root까지 Below 순회, 방문 Set으로 cycle 검사
  Root에서 Above 순회
  각 쌍의 양방향 링크 확인
  Captured Member 수 <= 1 확인
  Root만 Movement 활성인지 확인
  반환한 Count와 GetStackCount 일치 확인
```

스택 결합 직후, 분리/Destroy 직전후, 캡처 진입/해제 직후에만 실행해도 대부분의 구조 오류를 빠르게 잡을 수 있다.
