# 10. 리스크·미완성·테스트 체크리스트

## 1. 분석 기준

이 문서의 “확인”은 C++과 ini 정적 분석으로 재현 경로가 명확한 항목을 뜻한다. `.uasset` Blueprint 그래프나 레벨 인스턴스 연결에 따라 보완될 수 있는 것은 “에디터 확인 필요”로 구분했다.

우선순위 기준:

- P0: 실행/데이터 손상 등 즉시 차단
- P1: 핵심 게임플레이 의도와 실제 동작이 다르거나 진행을 막을 가능성
- P2: 특정 조건의 오동작, 유지보수 위험, 주요 미완성
- P3: 정리, 확장성, 낮은 빈도 문제

## 3. R-01 캡처 중 HP 미차감

### 근거

- `UCaptureComponent::HandleCapturedPawnAnyDamage`는 `OnCapturedPawnDamaged`만 호출한다.
- `AMonsterCharacterBase::OnCapturedPawnDamaged_Implementation`은 입력 스턴과 넉백만 수행한다.
- `AGoombaCharacter` 파생 구현도 스택 Root 넉백만 수행한다.
- 코드 주석은 “HP는 캡처 컴포넌트가 마리오로 라우팅”한다고 설명하지만 실제 HP 함수 호출이 없다.
- `AMarioCharacter::TakeDamage`는 캡처 중 직접 피해를 0으로 반환한다.

### 영향

굼바 스택, 일반 캡처 몬스터 등은 공격을 맞아도 마리오 체력이 줄지 않고 피격 모션/넉백만 발생할 가능성이 높다. 캡처가 사실상 무적 상태가 된다.

### 권장 방향

CaptureComponent가 Mario의 별도 `ApplyCapturedDamage` 같은 함수를 호출하게 하고, 그 함수는 일반 `TakeDamage`의 “캡처 중 무시” 가드와 분리해야 한다. 재진입/중복 피해를 막기 위해 Pawn의 원래 TakeDamage와 Mario 전달 경로가 한 번만 실행되는지 보장해야 한다.

### 재현 테스트

1. 굼바 캡처
2. 적 공격이나 킬러 충돌로 Captured Pawn에 1 피해
3. 피격 리액션과 HUD HP를 동시에 관찰
4. 예상 의도가 “마리오 HP 감소”라면 3→2가 되어야 함

## 4. R-02 보스 BGM 오케스트레이션

### 근거

`ABossArenaController`에 Start/Stop 함수는 있지만 C++ 조우 시작·컷신 종료·보스 스폰 경로에는 Start 호출이 없다. Stop은 승리와 사망에만 있다. `ABgmManager::SetSuppressed` 호출도 없다.

### 영향

- Blueprint/Level Sequence 이벤트가 빠지면 보스 BGM 미재생
- 보스 BGM을 시작해도 월드 BGM과 중첩 가능
- 지속 AudioComponent가 레벨 경계를 넘어 남을 가능성

### 에디터 확인

- `BP_ABossArenaController`의 EventGraph
- `LS_BossEncounter` Event Track
- Arena 레벨의 BgmManager 상태

## 5. R-03 난간 기능 미완성

`ELedgeActionState`에는 HangStart/HangLoop/MoveLeft/MoveRight/ClimbEnd가 있고 LedgeDetector도 생성되지만, 현재 Overlap에서는 로그 외의 위치 보정·MovementMode 변경·입력 전이가 구현되지 않았다.

난간 기능을 출시 범위에 포함한다면 다음이 추가돼야 한다.

- 상부 바닥과 전방 벽의 이중 Trace
- Capsule이 들어갈 공간 검사
- Hang 위치 Snap과 중력/이동 중지
- 좌우 이동 중 연속 공간 검사
- Climb 종료 위치와 Root Motion/Collision 동기화
- 캡처·피격·사망 때 강제 해제

## 6. R-04 영구 저장 부재

GameInstance는 레벨 이동에는 유지되지만 프로세스 종료에는 유지되지 않는다. 현재 저장되지 않는 값:

- 수집한 슈퍼문 ID/총합
- 코인
- HP
- Arena Intro 재생 여부
- 체크포인트
- 열린 진행 블록/플랫폼 상태

프로토타입 세션 플레이가 목표면 정상이다. 재실행 후 이어하기가 요구사항이면 `USaveGame` 기반 스키마와 버전 관리가 필요하다.

## 7. R-05 마리오 상태 이중화

예를 들어 그라운드 파운드는 다음 값들이 함께 맞아야 한다.

- `State.GroundPound`
- `bIsGroundPoundPreparing`
- `bIsGroundPounding`
- `bGroundPoundStunned`
- `bGroundPoundUsed`
- `bWaitingForPoundJump`
- `bPoundJumpConsumed`
- 두 개의 TimerHandle

캡처 진입/해제, 피격, 사망, 착지, 다이브 전환이 모두 이 값을 수동으로 리셋한다. 한 경로가 빠지면 입력 잠금이나 GravityScale이 남을 수 있다.

권장 방향은 Action 종료 책임을 `ExitGroundPound(Reason)`, `ExitWallAction(Reason)`처럼 모으고, bool을 enum에서 파생할 수 있게 줄이는 것이다.

## 8. R-06/R-07 보스 공간·소유권 결합

### 고정 공간

- 중앙 복귀: `(0,0,Z)`
- 반대편 Dash: 현재 위치 X/Y 부호 반전
- IceRain 중심: 월드 `(0,0)`
- 캡처 손 높이: World Z 1,500
- 실제 보스 기본 스폰: `(0,0,1035)`

Arena가 원점에서 이동하거나 서브레벨/복수 아레나가 생기면 패턴이 어긋난다. `ArenaRoot` 또는 Center SceneComponent의 Local Transform 기준으로 계산해야 한다.

### 전역 정리

ArenaController의 Cleanup은 `TActorIterator`로 월드의 모든 AttrenashinFist/IceShard/IceTile을 파괴한다. Owner나 Boss 연결로 범위를 좁히지 않는다. 복수 전투나 장식용 동일 Actor가 있으면 함께 제거될 수 있다.

## 9. R-08 승리 후속 처리

HeadHit 3회 뒤 C++는 보스/BGM/잔여물을 정리하고 프록시를 숨긴다. 다음은 보이지 않는다.

- 슈퍼문/코인 보상
- 승리 UI
- 퇴장 포탈 또는 문 개방
- 승리 컷신
- GameInstance 클리어 플래그

Blueprint가 이를 처리한다면 HeadHitCount Polling과 Destroy 시점을 고려해야 한다. 외부 Blueprint가 BossActor Destroy 후 값을 읽으려 하면 참조가 사라질 수 있다.

## 10. R-09 캡처 Pawn과 수집물

공통 수집물은 `OtherActor->IsA(AMarioCharacter)`만 허용한다. 캡처 중에는 PlayerPawn이 몬스터이고 마리오는 숨김/Collision Off이므로 코인, 하트, 슈퍼문을 획득할 수 없다.

캡처 상태에서 수집을 허용하려면 PlayerController/ViewTarget/CaptureComponent를 통해 원본 마리오를 Resolve하는 공통 함수를 써야 한다. 의도적으로 금지한 것이라면 레벨에서 캡처 전용 경로에 필수 수집물을 두지 않아야 한다.

## 11. 낮은 우선순위 구조 위험

### 동적 BGM Zone

BgmManager는 BeginPlay 때 Zone을 한 번만 수집한다. 런타임 스폰 또는 스트리밍으로 나중에 들어온 Zone은 폴링 목록에 없다.

### 하드 경로

`BP_Mario`, `WBP_HUD`, `WBP_StartScreen` 경로를 C++ ConstructorHelpers로 직접 로드한다. 폴더 이동 시 GameMode만 fallback하며 UI는 조용히 생성되지 않을 수 있다.

### 컷신 fallback

StartFlow는 명시 참조와 Tag를 모두 못 찾으면 첫 LevelSequenceActor를 재생한다. 레벨에 Sequence가 여러 개면 잘못된 연출이 실행될 수 있다.

### 템플릿 잔여

DefaultGame의 ProjectName, `AMarioOdysseyCharacter`, ThirdPersonMap 중복, StarterContent 자동 추가는 프로젝트 경계를 흐린다.

## 12. 기능별 회귀 테스트

### 이동

- 1/2/3단 점프가 0.25초 착지 연계 기준으로 전환
- 웅크린 속도 경계에서 LongJump/Backflip 정확히 분기
- Roll 중 역입력/낙하/피격/캡처 시 마찰과 MaxWalkSpeed 복구
- GroundPound 준비 중 Dive 전환 후 GravityScale/AirControl 복구
- PoundJump 창에서 1회만 발동
- WallSlide 진입 가능한 점프와 금지 액션 구분
- 같은 벽 연속 WallKick 시 재진입 떨림 없음
- 경사 12도 전후와 방향 내적 0.55 전후 부스트 경계

### 캡처

- 모자 조기 Release가 최소 Hover 0.5초를 지킴
- 최대 3초 후 강제 귀환
- 벽/모서리에서 귀환 정지하지 않음
- 캡처 성공 후 Pawn은 몬스터, ViewTarget은 Mario
- 해제 위치가 막힌 경우 NavMesh fallback
- 캡처 Pawn 파괴/사망 시 마리오 복귀
- 해제 후 1초 무적 후 정상 피격
- 캡처 중 피해가 기획된 HP 규칙과 일치

### 굼바

- AI 순찰→추격→탐색→복귀
- 캡처된 스택만 다른 굼바 위에 결합
- Top 캡처/Root 이동 일관성
- 중간/Root 파괴 시 연결 안전성
- 4단 스택으로 Goombette 보상 1회 생성

### 진행

- MoonId 중복 획득 차단
- Moon 10개에서 포탈 즉시 활성화
- Map→Arena 스폰 Transform과 페이드 1회 Consume
- 사망 체크포인트 복귀와 월드 상태 유지
- 특정 MoonId와 총 개수 AND 조건 플랫폼
- 레벨 재진입 시 이미 먹은 슈퍼문 제거

### 보스

- 일반 바닥 Slam은 기절하지 않음
- IceTile 위 Slam만 5초 Stunned
- Stunned 외 캐피는 캡처 실패/피해 처리
- 캡처 손 Barrage 3회 피격 시 강제 해제
- 1/2/3번째 머리 타격의 Phase2/Phase3/승리 전환
- Player Death가 보스/손/얼음/컷신/BGM을 모두 정리
- 재도전 때 프록시 Transform과 카메라 정상화
- 월드 BGM과 보스 BGM 중첩 없음

## 13. 자동화 권장 순서

1. 순수 수학/조건: Downhill 방향, Moon 조건, BGM Zone 우선순위
2. 상태기: GroundPound, Roll, Fist Slam, Boss Phase 흐름
3. Functional Test Map: 캡처 Possess/Release, Goomba Stack, Portal Travel
4. PIE 시나리오: 시작 화면 → 수집 → Arena → 보스 → 사망/재도전
5. Packaged Development 빌드에서 하드 경로, Cook 맵, 입력, Sequence 재검증

현재 Source에서 AutomationSpec/FunctionalTest 구현은 확인되지 않았다. 기능 간 결합이 큰 프로젝트라 위 순서의 최소 회귀 테스트를 추가하는 효과가 크다.
