# 07. 진행도와 월드 상호작용

## 1. GameInstance가 진행 허브인 이유

`UMarioGameInstance`는 레벨 전환과 Pawn Possess 교체에도 살아 있는 세션 객체다. 이 프로젝트에서는 다음 데이터를 보관한다.

| 데이터 | 기본값 | 유지 범위 |
|---|---:|---|
| CurrentHP / MaxHP | 미설정 / 3 | 레벨 전환 |
| Coins | 0 | 레벨 전환 |
| SuperMoons | 0 | 레벨 전환 |
| CollectedMoonIds | 빈 Set | 레벨 전환 |
| PendingSpawnTransform | 없음 | 다음 레벨에서 1회 Consume |
| PendingFadeOut | 없음 | 다음 레벨에서 1회 Consume |
| ArenaIntroPlayed | false | 게임 세션 |

이는 디스크 SaveGame이 아니다. 실행 파일을 종료하거나 GameInstance가 새로 만들어지면 모두 초기화된다.

## 2. 이벤트 모델

GameInstance는 네 개의 Dynamic Multicast Delegate를 제공한다.

| 델리게이트 | 구독자 예시 | 용도 |
|---|---|---|
| `OnHPChanged` | MarioPlayerController | HUD 체력 갱신 |
| `OnCoinsChanged` | MarioPlayerController | HUD 코인 갱신 |
| `OnSuperMoonCountChanged` | HUD, Portal, Platform, Block | 총 슈퍼문 조건 반응 |
| `OnSuperMoonCollected` | HUD, Platform, Block | 특정 MoonId 조건/획득 연출 |

수집물이나 마리오는 UI를 직접 찾지 않는다. 데이터를 GameInstance에 기록하면 PlayerController가 이벤트를 HUD Blueprint 이벤트로 중계한다.

## 3. HP

최초 BeginPlay에서 GameInstance에 저장 HP가 있으면 마리오가 읽고, 없으면 자신의 기본 MaxHP/CurrentHP를 기록한다. 이후 피격과 회복마다 `SetHP`가 값을 Clamp하고 `OnHPChanged`를 Broadcast한다.

HeartPickup은 수치 증가가 아니라 `HealToFull()`을 호출한다. 사망 후 부활도 MaxHP로 완전 회복한다.

HP가 GameInstance에 있으므로 `Map → Arena` 이동 때 현재 체력이 이어진다. 반대로 체크포인트는 Transform이 MarioCharacter 인스턴스에만 있어 OpenLevel을 넘는 영구 체크포인트로 쓰이지 않는다.

## 4. Coin

`ACoinPickup`은 `CoinValue` 기본 1을 GameInstance에 더한다. `AddCoin`은 0 이하 입력을 무시하고 총합을 0 이상으로 제한한다.

코인은 현재 HUD 표시 외의 소비처가 C++에 없다. 상점, 사망 패널티, 해금 조건은 미구현이다. CoinPickup은 일반 수집물처럼 획득 직후 Destroy되므로 레벨을 다시 로드하면 다시 배치된다.

## 5. SuperMoon

`ASuperMoonPickup`은 `MoonId`와 `MoonValue` 기본 1을 가진다.

### ID 처리

- 에디터에서 MoonId를 지정한 경우 그 값을 사용
- 비어 있으면 Actor의 `GetPathName()`을 ID로 자동 사용
- BeginPlay에서 이미 수집한 ID면 즉시 Destroy
- 새로운 ID만 `CollectedMoonIds`에 추가하고 카운트 증가
- `NAME_None`으로 직접 Collect하면 중복 방지 없이 값만 증가

경로명 자동 ID는 같은 레벨/배치가 유지되는 동안 편리하지만 Actor 이름이나 레벨 구조를 바꾸면 기존 세션 ID와 달라질 수 있다. 영구 저장을 도입한다면 명시적 안정 ID가 필요하다.

## 6. 공통 수집물 기반 클래스

`AMarioCollectibleBase`는 Coin, Heart, SuperMoon의 공통 동작을 제공한다.

- Sphere Overlap, Pawn 채널만 감지
- `AMarioCharacter`만 기본 Collector로 인정
- 중복 Overlap 방지 플래그 및 충돌 즉시 비활성화
- 2D 또는 위치 기반 수집 사운드
- Blueprint `BP_OnCollected` 연출 훅
- Yaw 회전과 Sin 기반 Bobbing
- 획득 처리 후 기본 Destroy

캡처 Pawn은 `AMarioCharacter`가 아니므로 기본 수집물과 겹쳐도 획득하지 못한다. 이것이 의도인지 테스트가 필요하다. 캡처 중에는 숨겨진 마리오가 Pawn에 Attach되어 있지만 마리오 충돌이 꺼져 있어 수집 Overlap도 발생하지 않는다.

## 7. 체크포인트

`ACheckpointFlagActor`는 Sphere Overlap, Mesh Overlap, Mesh Hit의 세 경로로 활성화를 시도한다.

인식 가능한 SourceActor:

1. 마리오 자신
2. `AMarioCapProjectile`이면 Owner인 마리오
3. 기타 Actor이면 Owner가 마리오인지 확인

활성화하면 Flag의 `RespawnPoint` Transform을 마리오에 저장하고 BlueprintNativeEvent `OnCheckpointActivated`를 호출한다. 기본적으로 한 번만 활성화한다.

따라서 캐피를 체크포인트에 맞혀도 활성화할 수 있다. 캡처 몬스터는 Owner 관계가 항상 마리오로 설정되는 구조가 아니므로 직접 접촉 활성화는 보장되지 않는다.

## 8. 사망과 리스폰

마리오의 죽음은 레벨 재시작이 아니라 같은 Pawn을 체크포인트로 텔레포트하는 방식이다.

```text
HP 0 / FellOutOfWorld
  → 입력·이동 잠금
  → 캡처 중이면 강제 해제
  → Death Montage
  → Fade To Black
  → Checkpoint Teleport
  → HP Full / 상태 초기화
  → Fade From Black
  → 입력 복구
```

레벨을 다시 로드하지 않으므로 일반 수집물, AI, 월드 이동 오브젝트는 사망 전 상태를 유지한다. 보스 영역만 `ABossArenaController`가 Mario의 GameOver를 감지해 별도로 초기화한다.

## 9. SuperMoonPortal

포탈은 기본 슈퍼문 10개를 요구하고 `Arena` 레벨로 이동한다.

### 활성화

- BeginPlay에서 GameInstance의 현재 슈퍼문 수 검사
- `OnSuperMoonCountChanged` 구독
- 조건 미달이면 Trigger Collision을 NoCollision
- 조건 충족 시 QueryOnly로 켬

### 이동

1. 중복 이동 방지 플래그 설정, Trigger OFF
2. Arena 목표 스폰 위치/회전을 PendingSpawnTransform에 저장
3. FadeOut 시간과 입력 잠금 여부를 PendingFadeOut에 저장
4. 현재 입력 잠금
5. 0.35초 검은 화면 FadeIn + 0.05초 Hold
6. `UGameplayStatics::OpenLevel("Arena")`
7. 새 레벨의 Mario BeginPlay가 Pending 값을 Consume
8. 지정 Transform으로 이동하고 0.35초 FadeOut 후 입력 복구

Pending 데이터는 Consume 방식이라 한 번 적용하면 flag가 내려간다. 같은 GameInstance 안에서 이후 일반 리스폰에 잘못 재적용되지 않는다.

## 10. ProgressBlockMover

슈퍼문 조건을 만족하면 한 번 이동하는 단순 진행 블록이다.

조건은 AND 관계다.

- `RequiredSuperMoonCount > 0`이면 총 개수 충족
- `RequiredMoonId != None`이면 해당 ID 수집
- 둘 다 지정하면 둘 다 충족

조건을 만족하면 현재 위치에서 `TargetOffset` 기본 Z +300으로 1.2초 동안 SmoothStep 이동한다. `bTriggerOnlyOnce` 기본 true로 재실행하지 않는다.

## 11. MarioMovingPlatform

반복 이동 플랫폼도 동일한 슈퍼문 개수/ID 조건을 구독한다.

| 옵션 | 동작 |
|---|---|
| `TargetOffset` | 기본 X +400, 로컬/월드 선택 |
| `MoveDuration` | 편도 기본 2초 |
| `bPingPong` | Start↔End 왕복. false면 End 도달 후 Start로 점프하는 루프 |
| `bEaseInOut` | SmoothStep 사용 |
| `bStartActive` | 조건이 없을 때 BeginPlay 자동 시작 |
| `bRandomStartPhase` | 여러 플랫폼 동기화를 피하도록 위상 랜덤화 |

진행 이벤트가 여러 번 오더라도 이미 활성 상태면 EndPoint를 계속 재계산하지 않도록 방어한다. 정지 후 재활성화하면 현재 배치 위치를 새 Start로 잡는다.

## 12. 진행 구조의 한계

- SaveGame 미구현: 종료 후 코인/문/HP 복원 불가
- 체크포인트는 레벨 이동 시 보존되지 않음
- Coin 재화 사용처 없음
- 수집물은 캡처 Pawn으로 획득 불가
- 자동 MoonId는 레벨 리팩터링에 취약
- ProgressBlock의 최종 상태는 GameInstance 조건으로 BeginPlay 때 재실행하지만, 이미 이동 완료 위치 자체를 저장하지는 않음
- 동일 레벨을 OpenLevel하면 SuperMoon은 사라지지만 Coin/Heart는 다시 생성됨

영구 진행도를 목표로 한다면 `USaveGame`에 Moon GUID, 코인, 체크포인트 ID, 월드 상태를 저장하고 GameInstance는 런타임 캐시/이벤트 허브로 유지하는 구성이 적합하다.
